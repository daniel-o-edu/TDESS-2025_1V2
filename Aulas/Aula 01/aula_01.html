Contextualização

O Desafio do QA

O Desafio do Semestre

Nosso Papel como QA

Vocês atuarão como uma equipe de desenvolvimento criando uma solução para uma feira de investidores. O risco é alto: investidores não toleram falhas críticas.

A Missão

O papel de vocês não é apenas "caçar erros", mas garantir que o produto entregue tenha valor, segurança e usabilidade. Se falhar na apresentação, o investimento é perdido.

Anos 60 e 70

Debugging vs. Testes

Antigamente, não existia o "testador". O próprio programador testava apenas para ver "se funciona" (debugging). O grande problema disso é o viés de confirmação.

  • O Caminho Feliz: O dev testa só o fluxo que sabe que funciona (usuário e senha corretos).
  • O que é ignorado: Senha errada, campo vazio, usuário inexistente, tentativas repetidas.
Para entender melhor

A Ilusão do "Funciona"

Testar apenas o "caminho feliz" é como verificar a segurança de uma casa testando apenas se a sua chave abre a porta.

Você confirmou que entra, mas esqueceu de checar se a janela dos fundos está aberta ou se qualquer chave mestra também abre a porta.

O código pode parecer "ok" para o desenvolvedor, mas está cheio de buracos do ponto de vista do usuário.

Conceitos Fundamentais

Definições e a Tríade

O que é um Bug? O que é testar? Entendendo a anatomia do erro.

Origem Histórica

9 de Setembro de 1947

A equipe de Grace Hopper trabalhava no Mark II em Harvard quando a máquina parou.

O Primeiro Caso Real

A causa foi uma mariposa (moth) presa em um relé. Eles colaram o inseto no diário de bordo: "First actual case of bug being found". Daí surgiu o termo Debugging (desinsetizar).

Glenford Myers

O que é Testar?

Conceito vital: Testar não é provar que funciona.

"Teste é o processo de executar um programa com a intenção de encontrar erros."

  • Provar que funciona: Leva ao viés de confirmação (escolher caminhos fáceis).
  • Encontrar erros: Você vira um investigador buscando "cantos escuros".
Princípios do QA

As Leis de Myers

1
O Teste "Bom"
É aquele com alta probabilidade de encontrar um erro ainda não descoberto.
2
O Teste "Bem-sucedido"
É aquele que encontra um erro, não o que passa "limpo". (Igual a um exame médico).
3
Independência
O desenvolvedor não deve testar seu próprio código (ponto cego psicológico).
Cadeia de Causalidade

Erro, Defeito e Falha

Não são sinônimos. É uma relação linear e progressiva.

Termo O que é? Onde acontece?
1. Erro Ação humana (equívoco). Na mente/ação do dev.
2. Defeito O "Bug" físico no artefato. No código (pode dormir lá).
3. Falha O comportamento inesperado. Em tempo de execução.
Exemplo de Código

A Calculadora Quebrada

Erro: Dev pensa que soma é asterisco (*).
Defeito: O código escrito errado.
function soma(a, b) {
  return a * b; // O defeito está aqui
}
A Falha

Se o usuário digitar 2 + 2, retorna 4. (Sem falha visível).
Se o usuário digitar 2 + 3, retorna 6 (Era esperado 5). Aqui ocorre a Falha.

Atividade Mental

Caso do Cupom de Desconto

Requisito: Desconto para compras estritamente ACIMA de R$ 100,00.

// O que o Dev escreveu (distraído):
if (valor_compra >= 100) {
  aplicar_desconto(0.15);
}
  • Ana (R$ 80): Pagou R$ 80. (Sem falha, apesar do defeito).
  • Bruno (R$ 100): Ganhou desconto indevido. FALHA!

Reflexão: A compra da Ana não ativou a falha porque o fluxo não passou pela condição errada.

Desastres de Software

Quando a Falta de Teste Mata

Knight Capital ($440 Milhões)

Código morto ("zumbi") foi ativado acidentalmente em produção. Comprou ações descontroladamente. Prejuízo de US$ 440 mi em 45 min.

Therac-25 (Vidas Humanas)

Condição de corrida (Race Condition). Se o operador digitasse muito rápido, a máquina irradiava o paciente sem proteção. Causou mortes.

Ariane 5 (Explosão)

Reutilização de código do Ariane 4. Overflow de inteiro (velocidade horizontal > 16 bits). Autodestruição em 37s.

Perguntas Comuns

O desenvolvedor não pode testar o próprio código?
Pode e deve (Unitários), mas ele tem o viés de confirmação. O QA independente é necessário para achar o que o dev não viu.
Todo defeito gera uma falha?
Não. Um defeito pode ficar "dormindo" no código e só gerar falha quando executado sob condições específicas (Ex: Compra da Ana).

Tipos de Teste e Prática

Classificações e Atividade

Caixa-Branca, Caixa-Preta e a tarefa para a próxima aula.

Classificações Iniciais

Caixa-Branca vs. Caixa-Preta

Caixa-Branca (Estrutural)
  • Visão: O testador VÊ o código fonte.
  • Foco: Lógica interna, loops, `if/else`.
  • Exemplo: Garantir que todos os ramos do código são executados.
Caixa-Preta (Funcional)
  • Visão: O testador NÃO vê o código. Age como usuário.
  • Foco: Entradas e Saídas (O que o sistema faz).
  • Exemplo: Clicar em "Calcular" e verificar o resultado na tela.
